Chapter 07

Modeling content in emdash

Unlike most Astro CMSes, emdash doesn't hard-code content types in TypeScript. You define them through the admin UI, and emdash handles the DB, migrations, and types.

Creating a content type

  1. Go to /_emdash/admin → Schemas → New Schema.
  2. Name it (e.g. product).
  3. Add fields — each has a type (text, number, boolean, rich text, reference, image, etc.) and options (required, default, validation).
  4. Save.

emdash creates the DB tables and exposes the new type through the API automatically. No migration file to write, no CREATE TABLE to run.

Regenerating TypeScript types

Whenever you change a schema, regenerate the type definitions so your editor stays in sync:

npx emdash types

After running this, getEmDashCollection('product') is fully typed, and autocomplete will show every field you defined.

Portable Text: the rich-text format

Rich-text fields store Portable Text — a structured JSON array, not HTML. That matters for three reasons:

  • No broken HTML from a user paste.
  • You can render the same content to different frontends (web, mobile, voice) from one source.
  • You can traverse the tree and apply custom behavior — auto-linking mentions, stripping disallowed marks, translating inline.

You usually don't interact with the raw JSON. The loader renders it to HTML for you. But when you need custom rendering, the tree is yours to walk.

Querying content from your pages

Top-level helpers you'll reach for most often:

import {
  getEmDashCollection,
  getEmDashEntry,
} from 'emdash';

// All entries of a type (destructure the result)
const { entries: allPosts } = await getEmDashCollection('post');

// Filtered + paginated
const { entries: published, nextCursor } = await getEmDashCollection('post', {
  status: 'published',
  orderBy: { publishedAt: 'desc' },
  limit: 10,
});

// One entry (returns { entry, error, isPreview, ... })
const { entry: post } = await getEmDashEntry('post', 'my-slug');
Return shapes. getEmDashCollection returns { entries, error, cacheHint, nextCursor }. getEmDashEntry returns { entry, error, isPreview, fallbackLocale, cacheHint }. Full filter / result types are in chapter 13 (API reference).

Seed data vs. live content

Each template ships a seed/seed.json that gets loaded by emdash seed. Treat it as initial content, not the source of truth:

  • On first bootstrap, seed data populates the database so your site has something to render.
  • After that, the database is the source of truth. Edits in the admin go to the DB, not to seed.json.
  • If you re-run emdash seed, it re-imports the seed (behavior depends on the template — some idempotently upsert, some reset).
A workflow that scales. Keep seed/seed.json under version control as canonical "example content" for new contributors, and let the production DB diverge freely. Cloudflare D1's export / backup tools let you snapshot prod content when you want.

Relationships between types

Fields of type reference link one content type to another (e.g. a post references a category). When you query a post, its references are resolved automatically, so you can access post.data.category.name without a second query.
