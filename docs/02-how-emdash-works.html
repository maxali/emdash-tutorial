Chapter 02

How emdash works

The one architectural fact that makes everything else click: emdash is not a standalone application. It's an Astro integration.

You install emdash into any Astro project. When you run that project, the integration:

  1. Mounts an admin panel at /_emdash/admin (React + TipTap editor).
  2. Creates a database schema, runs migrations, and exposes a content API.
  3. Registers an Astro live content collection called _emdash that your pages query.
  4. Optionally starts a plugin sandbox runner powered by Cloudflare's Dynamic Workers.

Three ideas to remember

1. Content lives in a relational DB, not in files

Most Astro CMSes store content as markdown files in src/content/. emdash is different. Your content is rows in D1 (or SQLite/Postgres), stored as Portable Text — a structured JSON format. The admin UI reads and writes those rows. Developers can still commit seed data (seed/seed.json), but the source of truth at runtime is the database.

2. Pages consume content through one live collection

You don't call the usual Astro getCollection("posts"). Instead you call getEmDashCollection("posts"), which queries the database through emdash's loader. Under the hood this is an Astro "live" content collection — new in Astro 6 — which lets collections resolve at request time instead of only at build time.

3. Plugins are sandboxed

Every plugin runs in its own V8 isolate and declares a capability manifest up front. A plugin that didn't declare write:content.page cannot write to pages — not in a sneaky way, not ever. This is the defining architectural difference from WordPress, where any plugin can DROP TABLE users.
