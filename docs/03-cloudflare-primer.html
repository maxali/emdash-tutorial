Chapter 03

Cloudflare in ten minutes

emdash is Cloudflare-first. If you've never used Cloudflare's developer platform, this chapter is for you. If you have, skip to the next chapter.

Sign up

Go to dash.cloudflare.com and create an account. No credit card is required for the free tier, which covers everything needed to try emdash:

  • Workers: 100,000 requests/day
  • D1: up to 10 databases, 500 MB each
  • KV: 100k reads, 1k writes per day
  • R2: 10 GB storage, zero egress fees at every tier
Plugins require the Paid plan. The only emdash feature that needs the $5/month Workers Paid plan is the sandboxed plugin system (it relies on Dynamic Workers / Worker Loader). Disable plugins and you stay on the free tier.

Cloudflare Workers

A Worker is a JavaScript / TypeScript function that runs on Cloudflare's global network. Unlike AWS Lambda (containers, 100–1000 ms cold starts), Workers run inside V8 isolates — the same lightweight sandbox Chrome uses for browser tabs. Cold start: under 5 ms.

Hello world is a single file:

export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    return new Response("Hello from the edge!");
  }
};

No Express, no framework. Request and Response are standard Web APIs.

The Wrangler CLI

Wrangler is how you build, run, and deploy Workers. It's an npm package:

npm install -g wrangler
wrangler login          # opens a browser OAuth flow

Commands you'll use constantly:

CommandWhat it does
wrangler devLocal dev server (powered by workerd, the real runtime)
wrangler deployPushes your Worker to Cloudflare's global network
wrangler d1 create <name>Creates a D1 database and prints its ID
wrangler r2 bucket create <name>Creates an R2 bucket
wrangler kv namespace create <name>Creates a KV namespace
wrangler secret put <KEY>Uploads an encrypted secret

wrangler.jsonc — the project config

Every Worker project has a wrangler.jsonc (or wrangler.toml). JSONC is the newer, recommended form — it supports JSON Schema validation in editors. A minimal emdash site's config looks roughly like this:

{
  "name": "my-emdash-site",
  "main": "./dist/_worker.js/index.js",
  "compatibility_date": "2026-04-20",
  "compatibility_flags": ["nodejs_compat"],

  "assets": { "binding": "ASSETS", "directory": "./dist" },

  "d1_databases": [
    { "binding": "DB", "database_name": "my-emdash-db", "database_id": "<paste-id-here>" }
  ],
  "r2_buckets": [
    { "binding": "MEDIA", "bucket_name": "my-emdash-media" }
  ],
  "kv_namespaces": [
    { "binding": "SESSIONS", "id": "<paste-id-here>" }
  ]
}

Bindings — the single biggest concept

A binding is how your Worker accesses Cloudflare services. You declare the relationship in wrangler.jsonc, and at runtime your code receives a ready-to-use client on the env object:

// env.DB        → D1 database client
// env.MEDIA     → R2 bucket client
// env.SESSIONS  → KV namespace client

There are no connection strings and no API keys in your code. The platform injects the credentials, and you can't accidentally leak them in a log.

The three storage services at a glance

ServiceWhat it isWhen to use
D1SQLite at the edge, with global read replicasRelational data — posts, users, comments. emdash's primary store.
R2S3-compatible object storage, no egress feesFiles — images, video, uploads. emdash stores media here.
KVEventually-consistent key-value storeRead-heavy, flat data — sessions, feature flags, config.
Rule of thumb: D1 for rows, R2 for files, KV for tokens and small config.

Pages vs Workers — the 2025 merge

Cloudflare used to have two products: Workers for logic and Pages for static sites. In 2025 they converged into "Workers with static assets." One Worker can now both serve your static HTML and run your SSR code. emdash uses this model: your Astro site builds to dist/, Wrangler uploads both the Worker script and the static assets in a single deploy.

Local dev and Miniflare

wrangler dev runs your Worker locally using workerd — the exact same runtime Cloudflare uses in production. Local D1 becomes a local SQLite file, local R2 becomes a folder on disk, local KV is in-memory. You can develop fully offline, and behavior closely mirrors production.

If you need to test against real production resources (e.g., a seeded D1 database), run wrangler dev --remote.
