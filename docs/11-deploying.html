Chapter 11

Deploying to Cloudflare

From wrangler login to a live URL. Assumes you chose the -cloudflare variant of a template.

Step 1 — Create the Cloudflare resources

You need a D1 database, an R2 bucket, and a KV namespace.

# From inside your project directory
wrangler d1 create my-emdash-db
# → copy the database_id it prints

wrangler r2 bucket create my-emdash-media

wrangler kv namespace create SESSIONS
# → copy the id it prints

Step 2 — Wire up wrangler.jsonc

Open wrangler.jsonc in your project and paste the IDs into the respective bindings:

{
  "name": "my-emdash-site",
  "main": "./dist/_worker.js/index.js",
  "compatibility_date": "2026-04-20",
  "compatibility_flags": ["nodejs_compat"],

  "assets": { "binding": "ASSETS", "directory": "./dist" },

  "d1_databases": [
    {
      "binding": "DB",
      "database_name": "my-emdash-db",
      "database_id": "<paste-d1-id>"
    }
  ],
  "r2_buckets": [
    { "binding": "MEDIA", "bucket_name": "my-emdash-media" }
  ],
  "kv_namespaces": [
    { "binding": "SESSIONS", "id": "<paste-kv-id>" }
  ]

  // If you are on the free plan, remove or comment out any
  // "worker_loaders" block — it requires the Workers Paid plan.
}

Step 3 — Run migrations against production D1

Your template's package.json includes scripts for preparing remote D1. The exact script name can vary (pnpm db:migrate, pnpm emdash:init:remote, etc.) — check your template's package.json. The general pattern is:

# Apply the emdash schema to your remote D1 database
pnpm bootstrap --remote    # template-specific: check package.json scripts
# Optionally seed production with starter content
pnpm seed --remote         # same caveat
Under the hood, these commands use Wrangler's D1 query API (wrangler d1 execute DB --remote --file=...) to run the generated migrations against your actual Cloudflare database.

Step 4 — Deploy

pnpm deploy

This runs astro build and then wrangler deploy. Within seconds your site is live on *.workers.dev (or your custom domain if configured). Wrangler prints the URL.

Step 5 — Set up the production admin

Visit https://<your-worker>.workers.dev/_emdash/admin. The setup wizard runs once, you register a passkey, and you're the admin.

Step 6 (optional) — Protect the admin with Cloudflare Access

If you want zero-trust SSO on top of emdash's built-in auth:

  1. In the Cloudflare dashboard, go to Zero Trust → Access → Applications → Add application.
  2. Create a Self-hosted application with path /_emdash/*.
  3. Add a policy, e.g. "Emails matching *@yourcompany.com".
  4. Pick an identity provider (GitHub, Google, one-time PIN via email).

Now only authorized users even reach the admin URL — requests are rejected at Cloudflare's edge before touching your Worker.
